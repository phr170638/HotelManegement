


一、代码功能亮点


1. 用户注册链路
   邮箱发送验证码 -> Redis 存储验证码 -> 校验验证码 -> 注册用户 -> 分配普通用户角色，用户模块与通知模块职责分离

2. 邮箱验证码通知模块
   `notification` 模块承接邮件发送，`MailServiceImpl` 与 `VerificationCodeServiceImpl` 分层明确

3. 登录链路
   登录支持基于 Spring Security + JWT 的身份认证，服务端能根据用户角色做后台权限控制，前端也有后台页面入口控制。

4. 用户签到功能
   使用 Redis Bitmap 记录签到天数，按月维度存储，空间开销小，读取“今日是否签到 / 本月累计签到 / 连续签到”等数据效率较高。

5. 支付 MQ 解耦
   支付成功后不是在支付服务里同步串行处理订单和通知，而是通过 RabbitMQ 发布支付成功事件，再由订单消费者和通知消费者分别处理。

6. 支付结果页主动对账
   除了依赖支付宝异步回调外，还补了服务端主动同步支付状态的能力，避免本地开发场景下因回调地址不可达导致结果页长期卡死。

7. 优惠券模块
   已实现兑换码领券入口、我的优惠券查询、优惠券状态转换、用户券快照存储等基础能力，业务闭环较完整。
   同时使用了分布式锁和数据库乐观锁：
   - 用户维度锁解决“同一个用户并发重复领券”
   - SQL 原子更新解决“券库存超发”
   锁相关代码通过工厂模式和策略模式封装，不是把 Redisson 调用硬写死在业务代码里，后续切换锁类型、超时策略、锁粒度会更方便。

8. 资源管理能力完整
    已具备国家、城市、酒店、房型、图片、设施等资源管理接口，前端后台页面也能直接维护酒店和房型基础信息。酒店搜索支持关键词、城市、星级、价格区间、评分、分页等条件，前端搜索页也实现了 URL 状态同步。页面按用户、订单、酒店详情、搜索、后台管理等维度拆分，API 封装统一，经由 `axios` 拦截器完成统一错误处理。

9. 测试覆盖已经成体系
    后端已有用户、订单、搜索、资源、评论、通知等模块测试。

二、Bug 与解决


1. 邮件服务对配置存在启动期硬依赖
   `MailServiceImpl` 使用 `@Value("${spring.mail.username}")` 强制注入邮箱账号。在本地有 `application-local.yml` 时不明显，但到了 CI 或无本地敏感配置的环境，Spring 在创建 `mailServiceImpl` 时会直接抛 `BeanCreationException`，进而导致整批测试的 `ApplicationContext` 加载失败。

   把邮件配置改成可缺省注入，例如 `@Value("${spring.mail.username:}")`，并在真正发信前增加配置校验：`fromAddress / host / port` 缺失时只记录 `warn` 并跳过发信，不阻塞应用上下文启动。这样邮件能力变成“可选基础设施”，而不是“系统启动硬前置条件”。

2. 开发环境配置对本地敏感配置存在覆盖问题
   `application-dev.yml` 中曾出现通过空默认值覆盖本地邮箱配置的情况，导致 `spring.mail.username` 最终被解析成空串。这个问题属于配置层优先级和占位符写法不当，而不是业务代码本身的问题。

   去掉会把本地真实配置覆盖掉的空默认值，改成“不提供时为空，但不覆盖已有值”的写法；同时保留 `application-local.yml` 只放敏感配置，保证开发环境公共配置与本地私有配置职责分离。

3. 支付结果强依赖支付宝异步回调，而回调地址默认是 localhost
   默认 `notify-url` 指向 `http://localhost:8090/...`，支付宝服务器无法访问开发者本机的 localhost，导致支付成功后异步通知根本回不来。前端结果页只认订单状态是否变成已支付，因此会长期停在“支付确认中”。

   增加服务端主动对账接口，在结果页轮询时补做“主动查询支付宝支付状态”。这样即使异步回调因本地环境不可达而失败，结果页仍有机会通过服务端主动同步把订单状态推进到正确值。与此同时，保留 RabbitMQ 异步主链路，不用同步逻辑替代模块解耦设计。

4. 数据库补字段脚本与 MySQL 版本兼容性差
   启动时用了 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` 这类并不是所有 MySQL 版本都支持的语法，导致某些环境启动时直接 `BadSqlGrammarException`，应用主进程退出。

    改为先查询 `information_schema.columns` 判断字段是否存在，再执行普通 `ALTER TABLE ... ADD COLUMN`.

5. 优惠券并发控制职责一度混在一起
    领券逻辑里既包含用户限领，又包含库存扣减，还包含事务创建用户券，如果所有并发控制都揉在一个方法里，后续难以解释“到底是悲观锁解决了什么，乐观锁又解决了什么”。

    把“用户维度分布式锁”和“库存乐观锁 SQL”职责拆开：
    - 分布式锁负责同一用户并发重复领取
    - SQL 原子更新负责库存超发
    同时用工厂模式和策略模式管理锁实现与超时策略。

6. JMeter 压测结果曾大量被脚本配置错误污染
    一开始的错误率并不代表后端扛不住，而是压测脚本本身有大量问题：登录接口误用 GET、Content-Type 错误、参数名不匹配、未带 token、订单接口用表单方式提交 JSON 等。

    先校正压测脚本，再看性能：
    - 登录接口统一用 `POST + application/json`
    - 下单接口统一用 `Body Data` 传 JSON
    - 后续带上从登录响应中提取的 token

7. 压测期间 SQL 控制台日志会严重污染性能数据
    `StdOutImpl` 在高并发下持续输出 SQL，会显著拉高 CPU 和控制台 IO，导致看起来像“系统性能差”，实际是在被日志系统拖慢。

    压测期关闭 MyBatis 控制台 SQL 输出，保留必要的业务日志。这样压测时看到的 CPU、吞吐和延迟才更接近真实业务瓶颈，而不是日志刷屏造成的假象。

8. CI 环境服务编排不完整
    GitHub Actions 最初只启动了 MySQL 和 Redis，没有启动 RabbitMQ，而测试上下文和消息监听容器会主动连接 `localhost:5672`，所以出现多条 `Connection refused`。

    在 CI workflow 中补齐 RabbitMQ service，并显式注入 `SPRING_RABBITMQ_HOST/PORT/USERNAME/PASSWORD`。这样测试环境和应用运行时依赖一致，避免“本地有 MQ、CI 没 MQ”的环境割裂。

9. 安全断言与当前实际行为不一致
    很多老测试断言“未登录返回 401”，但项目当前安全链路在这类场景下实际返回 `403`。

    统一按当前 Spring Security 实际行为更新断言，同时保留“无权 / 未认证”的场景说明，让测试与真实安全语义保持一致，避免误把“框架升级或配置变化”当成业务回归。
